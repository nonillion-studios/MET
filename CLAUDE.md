# CLAUDE.md ‚Äî MET (Manga Editing Tool)

Architecture guide for future Claude sessions working on this repo. Read this before touching `src/components/studio/`.

## What this is

**MET** is a React 19 + TypeScript + Vite + Konva web app for manga/manhwa cleaning, translation, and typesetting. It is a normal bundled SPA (not the single-file/offline architecture described in the orphaned prototype doc at `src/assets/claude.md` ‚Äî that file documents a *different*, disconnected reference prototype at `src/assets/mangastudio (1).html`; useful as design reference only, not part of the shipping app).

Entry point: `src/App.tsx`, rendered via `src/main.tsx` into `index.html`. Top-level navigation (`src/config/navTabs.ts`) has 5 tabs: Library, Text Editor, Settings, Teams, Cloud ‚Äî there is still no dedicated Home/landing screen (Recent/Templates/Tutorials/Plugins/Account), see "Known gaps" below.

## Data model

`src/types.ts` defines the persisted hierarchy:

```
Workspace { mangas: MangaSeries[] }
  MangaSeries { volumes: Volume[] }
    Volume { chapters: Chapter[] }
      Chapter { pages: Page[] }
        Page { original: ProcessedImage, cleaned: ProcessedImage | null }
```

`workspaces: Workspace[]` lives in `App.tsx` state, autosaved to IndexedDB (`idb-keyval`, key `workspaces_library`) debounced 800ms (`App.tsx`). Schema evolution goes through `src/lib/migrate.ts`'s `migrateWorkspace()`, run once on load.

Original/cleaned pairing: `src/lib/pages.ts`'s `suggestPairing()` ‚Äî three passes: exact normalized-filename match, then numeric-page-number match (extracts the trailing digit run from each filename), then positional fallback for whatever's left. Always returns an editable suggestion (`PairingSuggestion`), never a silent final assignment ‚Äî `PageManager.tsx` surfaces unmatched images as a manual drag-to-pair pool.

## The Studio (`src/components/studio/`)

`Studio.tsx` is the main shell, wrapped in three context providers (`ColorProvider`, `HistoryProvider`, `DockProvider`). Mounted only when a chapter is opened (`App.tsx`, `chapterView === 'studio'`); fully unmounts when navigating back to the page manager or a different chapter, so per-chapter state doesn't need explicit reset logic ‚Äî a fresh mount is a fresh chapter.

### Per-page layer stack

`layersByPage: Record<pageId, StudioLayer[]>` in `Studio.tsx` ‚Äî every page always has a locked `Background` layer at **root index 0**. `StudioLayer` (`studioTypes.ts`) covers `background | clean-patch | text | adjustment | group` types, and every one of them is implemented. (A phantom `bubble-mask` member used to sit in this union with no create path and no renderer; it was deleted rather than left as a stub to build against.)

**The stack is a tree, and `src/components/studio/layerTree.ts` owns its invariant** ‚Äî the same role `textRuns.ts` plays for runs. Groups carry `children`; nothing outside `layerTree` may walk or edit that field by hand, and `updateLayersOnPage` remains the only mutation primitive (every updater handed to it is composed from `layerTree`'s writers). Two rules it enforces: the background is root index 0 always, and `children` exists iff `type === 'group'`. **Illegal operations no-op rather than throw** ‚Äî cycles, moving the background, reparenting into a non-group. That's deliberate: the Layers panel computes drop intent from mouse geometry, so a bad drag must degrade to nothing happening, never to a corrupted tree. `layerTree.test.ts` covers all of it.

Because every new tree field (`children`, `mask`, `clipped`, `collapsed`) is **optional**, a pre-group (v4) save is already a valid tree ‚Äî every walker bottoms out immediately on it. That property is what keeps the v5 migration a no-op backfill; don't add a required tree field without rethinking it.

`cloneSubtree` returns an old‚Üínew `idMap` and callers **must** feed it to `StudioCanvasHandle.clonePaintCanvases`. The paint/mask registries are keyed by layer id, so duplicating a layer without cloning its canvas hands the copy a blank one ‚Äî that was a real, silent bug before `layerTree` existed (`handleDuplicateLayer` did `{...source, id: new}` and lost every pixel).

Raster pixel content for `clean-patch` layers lives **only** as live `HTMLCanvasElement`s in `paint/paintCanvasRegistry.ts`, held in a `useRef` inside `StudioCanvas.tsx` ‚Äî never in React state directly. This registry persists across page switches within a session (keyed by layer id, not page id), which is what lets `StudioCanvas.getExportSnapshot()`/`exportRasterLayers()` capture edits made on any page visited so far, not just the currently active one.

Real, working layer features: opacity, blend mode (`globalCompositeOperation` via `BLEND_TO_COMPOSITE`), visibility, lock, reorder, duplicate, delete. **Layer groups are real** ‚Äî create/ungroup (Layer menu, `Ctrl+G` / `Ctrl+Shift+G`, or the Layers panel), collapse, reparent by drag-and-drop, group opacity/blend with Photoshop's pass-through-vs-isolated semantics, and they survive PNG and PSD export (ag-psd gets real nested `children`, so a group stays a collapsible folder in Photoshop). **Adjustment layers** (`AdjustmentPanel.tsx`, `src/lib/adjustments.ts`) are real ‚Äî Brightness/Contrast, Hue/Saturation, and Levels ‚Äî and they **affect everything below them in their parent**, honouring their stack position. `layerTree.partitionAdjustments` rewrites each sibling list so every adjustment becomes a wrapper around the layers beneath it (`[A, B, adj1, C, adj2]` ‚Üí `adj2(adj1(A,B), C)`); the canvas renders each wrapper as a cached Group with `filters()` on it, and `exportImage.ts` walks the same partition. **Layer masks are real** ‚Äî see the dedicated paragraph below. **Not implemented**: smart objects. **Grouping refuses a selection spanning two parents** rather than silently reordering layers the user didn't select (`groupLayers`); the Studio surfaces that as a toast.

**Layer masks are real** (`LayerMask`/`StudioLayer.mask`, `studioTypes.ts`) ‚Äî any layer type may carry one, groups included, added/removed/enabled-toggled from the Layers panel (a mask thumbnail chip next to the layer's own icon, always visible ‚Äî click it to make the mask the paint target, click the layer's row to go back) or the Layer menu. Painting is **alpha-only**, not Photoshop's grayscale-luminance model: a brush/tool stroke reveals (raises alpha), the Eraser hides (lowers it), and soft brushes/low flow give feathered partial masks for free by reusing the *entire* existing paint engine unchanged ‚Äî the same "paint an alpha buffer with real tools" trick Quick Mask already relies on. Adding a mask seeds it from the active selection if one exists (`selectionToAlphaCanvas`, reused verbatim), otherwise fully opaque (reveal-all), matching Photoshop's "Add Layer Mask" default. Mask pixels live in their own registry (`maskCanvasRegistry` ‚Äî literally the *same* generic functions `paintCanvasRegistry.ts` already exports, reused against a second `useRef`, not a duplicated module) keyed by the mask's own id, mirroring how a layer's own raster pixels are kept separate.

**The mask trim is applied as a Konva `filters()` pass, not a `destination-in` sibling `KonvaImage` inside the cached group** ‚Äî that was the first implementation, and it went stale: a live-edited mask, once the layer's group was re-cached, could show the *pre-edit* mask forever after, because Konva's shape-level buffer-canvas draw path (`Shape.prototype.drawScene`, used when `_useBufferCanvas()` is true) re-derives a child's own composite operation relative to whichever node is currently being cached ‚Äî correct for a single cache pass, but not something to blindly re-derive on every keystroke of a paint stroke. A filter sidesteps the question entirely: `maskAlphaFilter(canvas)` reads the mask canvas's *current* pixels fresh on every invocation, and Konva's filter machinery already reruns cheaply (`getImageData -> filter -> putImageData` over the existing cached scene, no full redraw) ‚Äî the exact mechanism adjustment layers already prove out. An enabled mask forces the same isolation (caching) as an adjustment or a clip run, for the same reason: `filters()` only has something to run against once the node is cached. Editing a masked layer's *own* raster content (not its mask) needed one more fix: `suspendAncestorCaches`/`restoreAncestorCaches` (the "drop a cached ancestor for the duration of a stroke" mechanism that already existed for adjustments/clip-runs) only ever cleared *ancestor* caches, since an ordinary clean-patch layer never caches itself ‚Äî a masked leaf now does, so painting it needed the target's own cache included, not just its ancestors'.

PSD export converts the alpha-channel mask canvas to grayscale (`r=g=b=alpha`) at the PSD boundary only ‚Äî `LayerMask`'s own doc comment is explicit that alpha-channel storage is a live-rendering optimization, and ag-psd's `LayerMaskData.canvas` wants Photoshop's grayscale-channel convention instead. PNG/flatten export (`exportImage.ts`) mirrors the live canvas exactly: a masked leaf composites onto a scratch canvas, gets `destination-in`'d by the mask, then blits through the leaf's own opacity/blend ‚Äî same reasoning as the clip-run scratch-canvas pattern, just for export instead of Konva. `maskRaster` persists alongside `raster` in `SerializedStudioLayer`; both were already-optional v5 tree fields, so wiring masks up needed no schema bump.

Three non-obvious things about adjustments, all of which follow from "the wrapper *contains* the stack it adjusts":

- **Opacity is folded into the filter**, not applied to the wrapper (`adjustments.withStrength` lerps `original ‚Üí filtered`). Fading the wrapper would fade the *page* to transparent and expose the backing rect, not ease the grade.
- **Hiding is expressed by clearing `filters()`, never by `visible={false}`** on the wrapper ‚Äî hiding the wrapper blanks everything below it. (It did, briefly; `e2e/adjustments.spec.ts` caught it.)
- **Blend mode is not supported on adjustments and `LayersPanel` hides the control.** Photoshop blends an adjustment's result against its unadjusted backdrop, and the wrapper contains the very stack that backdrop would come from. Unlike opacity it isn't expressible as a per-pixel lerp. An inert dropdown would be worse than none.

**Blend modes were a no-op on screen until the Konva collapse, and nobody noticed** because `exportImage.ts` composites onto a single canvas and applied them correctly the whole time ‚Äî screen and export silently disagreed. Cause: `Container.drawScene` (`node_modules/konva/lib/Container.js`) applies `globalCompositeOperation` while drawing children **into the current canvas**, and every StudioLayer used to own a separate Konva `Layer`, i.e. a separate canvas that starts empty. Multiply against nothing returns the source. `e2e/compositing.spec.ts` pins this with exact arithmetic (grey√ógrey must be 64, not 128).

### Canvas engine (`StudioCanvas.tsx`)

**One Konva `Layer` holds the page and its entire stack; each StudioLayer is a `Group`** (`renderLayerNodes`, recursive). Konva `Layer`s are separate canvas elements and cannot nest, so the old layer-per-StudioLayer could express neither groups nor a working blend mode. Consequences worth knowing before editing the render:

- `visible`/`listening` are set from each layer's **own** flags ‚Äî Konva cascades both down a Group, so a hidden or locked group deafens its whole subtree for free. `layerTree`'s `isEffectivelyVisible/Locked` exist for consumers that *don't* get that free (the exporter).
- `Group` has **no `batchDraw`** (only `Layer` does) ‚Äî redraws go through `redrawLayerNode`, which hops to `getLayer()`.
- The page backing `Rect` (drop shadow) and the View Original `overlayImage` are **chrome, not content**: they sit outside the layer tree, between the background and the stack. Keep them out of any future group cache.
- **Group isolation is a two-renderer contract.** `needsIsolation` (`StudioCanvas.tsx`) and `isolatesGroup` (`exportImage.ts`) implement the same rule for the canvas and the exporter and **must not drift** ‚Äî when they did, the screen showed 96 and the file said 128 for the same group. The rule mirrors Photoshop: opacity 1 + normal blend is *pass-through* (children composite against everything below the group, which is what an uncached Konva Group already does for free); anything else is *isolated* (the subtree composites against nothing but itself, then blits once through the group's alpha/blend, via `cache()` / an offscreen canvas). A Multiply child inside an isolated group therefore washes out ‚Äî correct, and what Photoshop does. **Child count is not a condition**: isolation changes what a child blends *against*, not just how results stack, so "skip the cache for single-child groups" is a real bug, not an optimization. `e2e/export.spec.ts` compares decoded export pixels against a stage sample to hold this.

Konva `Stage` per page. Background image loads into `image` state from `page.original` or `page.cleaned` (via `showCleaned`); when `overlayOpacity > 0` and `showCleaned` is true, `page.original` also loads as a second `overlayImage` rendered on top at that opacity (View Original overlay mode).

Pan: Pan/Select tools drag the Stage natively; **any other tool** pans via Space-hold or middle-mouse-drag, handled manually through `panRef` + window-level `mousemove`/`mouseup` listeners (not Konva's built-in drag, to avoid fighting per-tool pointer handlers).

Tool routing in `handlePaintPointerDown/Move/Up` dispatches by `activeTool` string against several disjoint sets: `MARQUEE_TOOLS` (includes `crop`, which reuses the same rect-drag as Rectangular Marquee), `LASSO_TOOLS` (drag-based freehand only ‚Äî `lasso-polygon` is click-accumulate, handled separately via `handleStageClick`/`lassoPolyPoints`, mirroring the Pen tool's `penPoints` pattern), `PAINT_TOOLS` (from `paint/usePaintLayer.ts`). Marquee Shift constrains to a perfect square/circle; Shift/Alt held at the start of a marquee/lasso/wand drag instead means add/subtract, see Selection model below.

**New clean-patch layers are seeded with a copy of the background** (`Studio.tsx`'s `handleAddLayer` ‚Üí `StudioCanvas`'s `seedLayerWithBackground`), not left blank ‚Äî matches the standard manga-cleaning workflow ("duplicate the scan, clean the duplicate") and is what makes Clone/Heal/Blur/Sharpen/Smudge/Dodge/Burn/Sponge/Content-Aware/Liquify actually useful the moment a layer is created, since those tools only ever read/write the active layer's own canvas, never the (immutable, non-paintable) background underneath it. Relatedly, **picking any paint-family tool while a non-`clean-patch` layer is active** (e.g. the default Background layer on a freshly opened chapter) **auto-switches to the topmost existing raster layer, or creates one** (`Studio.tsx`, a `useEffect` keyed on `activeTool`) ‚Äî without this, every brush/fill/shape/liquify tool would silently no-op on first use, since `getActivePaintCanvas()` only returns a canvas for `clean-patch` layers.

**Magic Wand samples the background composite as a fallback** (`usePaintLayer.ts`'s `getFallbackCanvas`, wired to `StudioCanvas`'s `sampleCanvasRef` ‚Äî the same hidden canvas the Eyedropper already used) when no clean-patch layer is active, instead of silently operating on a nonexistent/blank raster canvas ‚Äî otherwise the wand could never select regions of the actual manga art, only whatever happened to be painted on an overlay layer.

**Crop is real** (`StudioCanvas`'s `commitCrop`, triggered by Enter or double-click while the Crop tool is active over a rect selection ‚Äî mirrors the existing Pen/Polygonal-Lasso "build then Enter/dblclick to commit" convention): trims the background (`page.original` + `page.cleaned`) and every raster layer's canvas to the selection rect, shifts text layers to match, and persists the new page dimensions back up through a new `onPagesChange` prop threaded from `App.tsx`'s existing `handleChapterPagesChange` (previously Studio had no way to mutate page image data at all). Only rectangular selections are supported, matching Photoshop's Crop tool.

**Zoom tool is real**: click zooms in centered on the cursor, Alt-click zooms out ‚Äî previously selecting it and clicking the canvas did nothing (zoom only ever worked via the scroll wheel or toolbar buttons, regardless of active tool).

All of the above dispatch through **Pointer Events** (`onPointerDown/Move/Up`), not separate mouse/touch handlers ‚Äî mouse, touch, and pen input share one code path. Real stylus pressure (`PointerEvent.pressure`) scales brush/pencil/eraser size (only for `pointerType === 'pen'`; mouse/touch report a flat, meaningless 0.5 per spec). A `touchCount >= 2` guard skips tool dispatch for pointer events fired mid-two-finger-gesture, deferring to the separate pinch/pan `onTouchMove` handler. Two-finger touch pans *and* pinch-zooms simultaneously (`handleTouchMove` anchors on the *previous* frame's pinch center, not the current one ‚Äî a plain two-finger drag with no distance change still pans correctly).

Selection model: `paint/selection.ts`'s `Selection` union (`rect | ellipse | polygon | mask`) ‚Äî vector shapes clip via `Path2D`, magic-wand masks approximate live via bounding box during a stroke and get pixel-perfect refinement on commit (`refineMaskedRegion`, now wired into every paint-stroke commit path, not just magic-wand). **Feather/expand/contract/add-subtract/intersect are real**: Shift(add)/Alt(subtract)/Shift+Alt(intersect) combine the in-progress marquee/lasso/wand shape with the prior selection by rasterizing both to a mask and compositing (`combineSelections`/`combineModeFromModifiers`); a **Select** menu (Deselect, Feather‚Ä¶, Expand‚Ä¶, Contract‚Ä¶, Transform Selection, Make Selection from Path, Quick Mask) applies one-shot pixel-amount ops (`featherSelection`, `growSelection`) via a `swal` numeric prompt. **Magnetic Lasso, Healing Brush, Patch Tool, and Slice are all real** ‚Äî see Part H tool descriptions elsewhere in this doc (heal/patch share Clone Stamp's alt-click source-point plumbing with color-matched blending; Slice queues rect selections and exports one cropped PNG per rect as a zip; Magnetic Lasso snaps each clicked segment onto nearby strong edges via Dijkstra over a Sobel gradient-magnitude cost field). **A real vector Pen Tool is implemented** ‚Äî see "Vector paths (Pen Tool)" below.

A marquee/lasso gesture doesn't touch `selection` at all until the pointer clears a small (4 screen px) dead zone (`marqueeDraggedRef`/`lassoDraggedRef`, `StudioCanvas.tsx`) ‚Äî below that it's an ordinary click (most commonly: starting a paint stroke *inside* an existing selection), not an attempt to draw a new empty one. Below the dead zone, pointerup is a pure no-op: nothing was ever touched, so there's nothing to revert either. Deselecting is otherwise only ever explicit ‚Äî `Ctrl/Cmd+D`, the Deselect button in the marquee/lasso/wand tool's options bar, or Esc while a plain selection tool (not Crop/Slice, which use Esc for their own draft-cancel) is active with a selection present.

**Selections are per-page.** `Studio.tsx`'s `selection` state is a single flat value, but switching `activePageId` stashes the outgoing page's selection into `selectionByPageRef` (mirroring how `rasterByPageRef`/`maskByPageRef` already keep other per-page data) and restores it if you come back ‚Äî it's never just reset to none. Quick Mask's scratch alpha canvas (`quickMaskCanvasRef`, a flat non-page-keyed ref inside `StudioCanvas`) and Multi-Bubble/Slice's queued rect arrays (`multiBubbleRects`/`sliceRects`, `Studio.tsx`) are **not** page-scoped the same way ‚Äî a page switch commits Quick Mask into the outgoing page's stashed selection (same as toggling it off) and clears the queued rects outright, so neither leaks pixels/regions onto whichever page you switch to next.

### Vector paths (Pen Tool)

The Pen and Curvature Pen tools build a real, persisted `path`-type `StudioLayer` (`PathLayerData` ‚Äî `anchors: PathAnchor[]` with per-anchor `point`/`handleIn`/`handleOut` offsets and a `type: 'corner'|'smooth'` hint, plus `closed`/`fill`/`stroke`/`fillRule`) instead of the old rasterize-and-discard behavior. Anchor placement is a real mousedown/mousemove/mouseup gesture in `StudioCanvas.tsx` (`penDraft`/`penPlacingRef`), not Konva click events: a plain click places a corner anchor, a click-and-drag places a smooth anchor with handles pulled out symmetrically along the drag vector (mirrors Photoshop exactly), and clicking back on the first anchor (with ‚â•3 already placed) closes the path. Curvature Pen's plain clicks still produce smooth curves without needing a drag ‚Äî `pathGeometry.ts`'s `applyCurvatureSmoothing` retroactively recomputes the *previous* anchor's handles (a Catmull-Rom-to-bezier tangent) each time a new anchor is appended, since a smooth anchor's handles depend on both neighbors and the anchor just placed doesn't have its next one yet.

**One shared function draws every path anywhere it appears**: `pathGeometry.ts`'s `traceAnchors(ctx, anchors, closed)` does real per-anchor `bezierCurveTo` tracing (genuinely different math from `paintEngine.ts`'s older `tracePenPath`, which is a polyline-smoothing approximation with no independent handle concept, left in place only for anything else still calling it) ‚Äî against a `Konva.Context` for the live pen preview and `PathLayerNode`'s committed render, or a real `CanvasRenderingContext2D` for export/bake, since both implement compatible `moveTo`/`bezierCurveTo` signatures.

**Path Selection (`path-select`) and Direct Selection (`direct-select`) are both real, and deliberately separate tools** rather than reusing the plain `select` tool ‚Äî a path has no `x`/`y` (its geometry *is* the anchor list), so dragging it means translating every anchor by the drag delta, a structurally different update than `select`'s "set `x`/`y`" contract. Path Selection's whole-path drag doesn't participate in the Transformer at all (no bounding-box resize, matching real Photoshop) ‚Äî `PathLayerNode`'s Group resets its own position to `(0,0)` after every drag rather than accumulating a transform, the same principle `TextLayerNode.onTransformEnd` already applies to scale. Direct Selection renders per-anchor/per-handle drag chrome only for the selected path: dragging an anchor moves just its `point` (handles ride along automatically, since they're stored as offsets); dragging a handle patches that one side and mirrors the opposite handle if the anchor is `type: 'smooth'`; **Alt-click an anchor converts it between corner and smooth** (`pathGeometry.ts`'s `toggleAnchorType` ‚Äî smooth‚Üícorner retracts both handles to zero, corner‚Üísmooth synthesizes symmetric handles from the anchor's neighbors), matching Photoshop's real gesture (there's no separate Convert Point tool).

**Stroke Path / Fill Path** (Layer menu, disabled without both an active path layer and an active raster layer) one-shot bake the path's geometry onto the raster layer's canvas via `paintEngine.ts`'s `strokePathOntoCanvas`/`fillPathOntoCanvas` (co-located there rather than in `pathGeometry.ts` because `pathGeometry.ts` must stay DOM-free ‚Äî it's covered by `pathGeometry.test.ts`, and `paint/selection.ts`'s `clipToSelection` has a module-scope `document.createElement('canvas')` that crashes vitest's DOM-free unit-test environment the moment anything imports it, even transitively). **Select > Make Selection from Path** (`paint/selection.ts`'s `pathToSelection`) flattens the path's bezier curves into a dense point list (`flattenPathToPolygon`, standard cubic-bezier subdivision ‚Äî polygon selections have no bezier concept of their own) and wraps it as an ordinary `{kind:'polygon'}` selection, which already does everything downstream for free.

Export: `exportImage.ts`'s flattened PNG/JPG/WEBP export rasterizes a path via `traceAnchors` (correct ‚Äî flattened export has no vector concept regardless of format, same as text glyphs). `exportPsd.ts` writes a **real vector path** into the PSD (`vectorMask`/`vectorFill`/`vectorStroke`, not a rasterize fallback) ‚Äî `ag-psd` genuinely supports Photoshop's native path-knot format, confirmed by an actual `writePsd`/`readPsd` round-trip during implementation: each `BezierKnot.points` is `[inHandleX, inHandleY, anchorX, anchorY, outHandleX, outHandleY]` in absolute page-pixel coordinates (`ag-psd` normalizes by width/height internally on write/read, so no manual scaling is needed on this side).

**A real, non-obvious gesture-timing bug found and fixed while building this**: Konva's Stage synthesizes a `dblclick` event from any two `click` events that land within its own timing window ‚Äî **regardless of where on the stage they landed**. Since Pen anchor placement happens on pointerdown/up (not click), two ordinary clicks placing two *different* anchors in quick succession (a completely normal fast-clicking usage pattern, not just an automated-test artifact) could misfire Konva's synthesized dblclick and prematurely finish the path after only two anchors. `handleStageDblClick`'s pen branch now only treats it as a real double-click-to-finish if the last two placed anchors are actually at (near) the same spot ‚Äî what an actual double-click always is ‚Äî otherwise it's a no-op and the path stays open for more anchors.

**Explicitly out of scope**: path-layer masks (raster alpha clipping vector paint ‚Äî the Layers panel's "Add Mask" action is gated against `type === 'path'` rather than silently no-opping), path-layer clipping (as a clip base or follower ‚Äî `canBeClipBase` stays raster-only), path participation in the adjustment stack (nothing to filter until baked), Free Transform Path (bounding-box resize/rotate of a whole path), and compound multi-subpath paths (`PathLayerData.anchors` models exactly one open-or-closed subpath; `fillRule` is included forward-compatibly for whenever that lands).

**The Move tool drags pixel content trapped inside an active selection** on the active clean-patch layer (`StudioCanvas.tsx`'s `movingSelectionRef`): a pointerdown that lands inside the selection's own shape (`selectionContainsPoint`, a real point-in-path/mask test, not just the bounding box) cuts that region out of the layer's canvas into a scratch `pieceCanvas` and redraws it at the drag offset each move, restoring only the shrinking union of old/new rect each frame rather than the whole canvas; `onSelectionChange` re-fires every move too (`translateSelection`), so the marching ants track the moved content live instead of jumping at drag-end. A press-and-release with zero movement reverts the cut and skips history ‚Äî otherwise every stray click inside a selection would count as a no-op edit. This is layer-content movement, not the layer-repositioning `x`/`y` drag the Canvas selection section below describes for text layers ‚Äî it works today specifically because raster pixels are a canvas to redraw, where a text layer only has a Konva node to reposition.

**Select > Transform Selection** (menu item, `Studio.tsx`'s `transformingSelection` state) drops a free-transform box (Konva `Rect` + `Transformer`) around the selection's bounds; dragging/resizing/rotating it and committing with Enter (Escape cancels) reshapes the selection's own geometry via `transformSelectionMask` ‚Äî an affine rotate+scale+translate around the box's pivot, rasterized through a canvas 2D transform since rotation/non-uniform scale can't be expressed by the vector `Selection` kinds. It **never touches pixel content**, only which pixels a subsequent paint stroke would be clipped to ‚Äî mirrors Photoshop's Select > Transform Selection exactly. While it's active, all other marquee/lasso/paint pointer dispatch is suppressed so only the box's own Konva drag/Transformer responds.

**Quick Mask** (`Q`, or Select menu, `Studio.tsx`'s `quickMaskActive`) redirects every paint tool onto a scratch alpha-only canvas (`quickMaskCanvasRef`) instead of the active layer, shown as a red rubylith tint (`redrawQuickMaskOverlay`, built with a `destination-out` punch of the mask's alpha out of a flat red rect ‚Äî no per-pixel JS loop needed even on live per-stroke redraws). Entering seeds the buffer from the current selection (`selectionToAlphaCanvas`); toggling off reads the buffer's alpha back into a real `Selection` (`alphaMaskToSelection`) via `StudioCanvasHandle.commitQuickMask()`. Painting in Quick Mask is deliberately unclipped by the outgoing selection (it's redefining it), so any tool ‚Äî brush, eraser, gradient, shapes ‚Äî becomes a selection-painting tool for free, the same "reuse the whole paint engine" trick `alphaMaskToSelection`'s doc comment calls out.

### Brush engine (`paint/brushTip.ts`, `paint/brushThumbnail.ts`, `BrushesPanel.tsx`)

Stamps are cached tip canvases (`getBrushTip`), keyed by size/hardness/shape/angle/roundness/colour (+ `maskId` for image tips). A dense stroke lays hundreds of stamps, so building the tip once and blitting it beats a per-stamp `createRadialGradient`+`arc`, and it's what makes non-round tips and imported image brushes possible at all. **Tips are rendered in their final colour, not tinted white-then-recoloured** ‚Äî a `source-in` tint can only be applied to a whole canvas, so tinting the dirty sub-rect of a page-sized stroke buffer would wipe the rest of the stroke.

**Flow and opacity are genuinely separate**, Photoshop-style: `usePaintLayer` accumulates a whole stroke into a scratch buffer at `flow`, then composites that buffer onto the layer at `opacity`, restoring only the dirty rect (`compositeStroke`). Painting straight onto the layer at `opacity * flow` ‚Äî which is what this used to do ‚Äî lets overlapping stamps within one stroke blow past the opacity cap, making the two sliders indistinguishable. Also real: spacing, angle, roundness (elliptical/calligraphic tips), scatter, pull-string smoothing, square tips, and pressure independently driving size and/or opacity.

`BrushCursor.tsx` draws the live outline (size x zoom, angle/roundness, dashed inner ring at a soft tip's falloff, crosshair under 4px) as **SVG above the Konva stage**, so it can never reach exports or the layer stack. The OS cursor is hidden while a brush-sized tool is armed.

Brush presets (`src/lib/brushStore.ts`, idb-keyval `brush_presets`, schema-versioned) are plain parameter sets over that one engine ‚Äî there is deliberately no separate "brush type", so an imported or duplicated brush is exactly as capable as a built-in. Built-ins live in code and are re-merged on load (only their `favorite` flag persists), so shipping a new one needs no migration. Panel thumbnails are drawn by the **real engine** (`renderBrushThumbnail` calls `strokeSegment`), so spacing gaps, scatter and pressure taper all show up truthfully. Imported images are baked to an alpha mask once at import (`imageToBrushMask`, alpha = 255 - luminance, existing alpha respected) rather than re-derived per stamp. **Favorites is a virtual folder** ‚Äî a view over the flag, not a location.

**Note:** reading `e.target.files` and then setting `e.target.value = ''` before consuming it silently drops the upload ‚Äî `FileList` is live, so clearing the input empties it. Snapshot with `Array.from(...)` first. This had already broken font upload in `FontsPanel`.

**Brush symmetry is real** (`PaintSettings.symmetry`, a Symmetry dropdown in the tool options bar for Brush/Pencil/Eraser): `usePaintLayer.ts`'s `applyStrokeSegment` mirrors each *segment* (not a post-hoc flip of the finished stroke) across the canvas center ‚Äî `horizontal` flips left-right, `vertical` flips top-bottom, `both` draws all four. Only mirrors around the canvas center, not an arbitrary user-placed axis, and only for the three raw-stroke tools (not clone/filter-brush/liquify) ‚Äî a real, honestly-scoped subset of what "brush symmetry" can mean.

**Liquify** (`paintEngine.ts`'s `liquify()`) is real ‚Äî `push`/`swirl`/`pinch`/`bloat`/`crystalize`/`reconstruct` modes. The first five each compute a per-pixel *source sample offset* (a true warp, not a blend) with radial falloff, same getImageData/putImageData-over-a-bounding-box pattern as the filter brushes. `reconstruct` blends back toward a pristine pre-liquify snapshot instead ‚Äî `StudioCanvas.tsx`'s `liquifySnapshots` registry captures one lazily per layer, on that layer's first-ever liquify edit (mirrors `paintCanvasRegistry`'s per-layer-id-not-page-id pattern), and it's cleared when the layer is deleted.

**Gradient** tool now goes foreground‚Üíbackground (`settings.bgColor`), matching Photoshop's default convention, instead of the old hardcoded foreground‚Üítransparent-white.

Grid (`showGrid`) and rulers (`showRulers`) are real, toggled from the View menu ‚Äî grid is a Konva overlay layer at a fixed 100px page-space spacing; rulers are HTML overlays tracking `pos`/`scale` with tick labels every 100px.

### Persistence

Two separate IndexedDB stores, deliberately kept apart so painting never triggers a full-library rewrite:

- `workspaces_library` (existing, `App.tsx`) ‚Äî chapter/page structure + image data URLs.
- `studio_<chapterId>` (`src/lib/studioProjectStore.ts`) ‚Äî layers (with raster pixels as data URLs), TypeR script/styles. Autosaved 1.2s after the last layer/TypeR change *or* committed paint stroke (`Studio.tsx`'s `scheduleAutosave`/`flushAutosave`). Loaded on mount; raster layers hydrate lazily as each page is visited (`loadRasterLayer` polls briefly for the background image to finish loading ‚Äî see `waitForImage` in `StudioCanvas.tsx`).
- `studio_versions_<chapterId>` ‚Äî capped (10) full-copy version snapshots pushed on every autosave. No diffing/compaction; fine at this scale, revisit only if real usage shows storage bloat.
- `text_styles` (`src/lib/textStyleStore.ts`) ‚Äî saved character/paragraph text styles. App-wide, not per-chapter (a style is meant to be reused across projects), which is why it isn't folded into `studio_<chapterId>`.

`studio_<chapterId>` is at **schema v5** (`STUDIO_SCHEMA_VERSION`; v2 added `autoWidth`/`letterSpacing`/`shadow`, v3 added `gradient`, v4 added `runs`, v5 made layers a tree). `migrateStudioLayers` runs on *any* mismatched version and every backfill in it is an idempotent `??`, so one pass takes v1‚Äìv4 straight to current ‚Äî don't restructure it into a per-step chain without keeping that property. This matters because the renderer/panel read nested fields (`text.shadow`, `text.gradient`) directly, so an un-backfilled layer throws on open. v5 itself adds **no** backfill: its fields are all optional and absent *is* the pre-v5 behaviour, so `migrateLayer` only had to start recursing.

`SerializedStudioLayer` nests (`children?`) and carries pixels as `raster` (the layer's own) and `maskRaster`. The load-time strip and the autosave merge (`Studio.tsx`) both go through `layerTree`'s `mapTree`/`flattenTree`, which are generic over the node shape precisely so serialized layers reuse them instead of growing a second walker that could drift. The registries stay **flat, keyed by layer id** ‚Äî nesting never reaches them.

Native project format: `.msp` (zipped JSON containing the full workspace tree + every chapter's studio data), `src/lib/mspFile.ts`. Export/import UI lives in `App.tsx`'s workspace list (not inside Studio, since it operates on a whole workspace).

### Export (`src/lib/exportImage.ts`, `src/lib/exportPsd.ts`)

- PNG/JPG/WEBP: `StudioCanvasHandle.getExportSnapshot()` captures background + full layer stack (raster layers as data URLs, text layers as structured data) for the active page; `compositeFlattenedImage()` flattens onto a canvas respecting opacity/blend/visibility, rendering text layers via canvas 2D (`fillText`/`strokeText`, with manual greedy word-wrap to approximate Konva's auto-wrap). JPG flattens onto white first (no alpha channel). `drawTextLayer` derives the text's **real** box before drawing ‚Äî point text (`autoWidth`) is never wrapped and is measured, since its stored `width` is not its actual size; that box then drives wrapping, centring *and* the gradient vector, so all three stay consistent with the canvas.
- PSD: `exportPsd()` builds an `ag-psd` `Psd` object ‚Äî one layer per `StudioLayer`, raster layers get a canvas, text layers get `LayerTextData` (editable in Photoshop; font family names pass through as-is, Photoshop substitutes if not installed ‚Äî can't resolve that from a browser), per-character runs map onto its `styleRuns` so mixed-size/colour text stays editable rather than flattened, and gradient text adds a `gradientOverlay` effect (see Text layers above). `ag-psd` is **dynamically imported** (`await import('ag-psd')`) so it code-splits into its own chunk instead of bloating the main bundle (it's ~300KB alone) ‚Äî its *types* are safe to import at top level, since `import type` is erased.
- `ExportDialog.tsx` wires both into the Project menu / `Ctrl/Cmd+E`.

Text export (TXT/DOCX/PDF) lives in `src/lib/textEditorExport.ts` ‚Äî see the Text Editor section below.

### Canvas selection (Select tool)

`Studio.tsx` holds `selectedLayerIds: string[]`; **`activeLayerId` is derived from it** (the last member ‚Äî the "primary"), not stored separately, so the two can't disagree. The single-layer panels (Text, Adjustment), the layer shortcuts and the edit target all key off the primary; the set drives multi-layer operations.

- Click selects, **Shift/Ctrl-click toggles** (canvas *and* Layers panel), click on empty canvas deselects.
- **Drag on empty canvas marquee-selects** objects. This is why only the Pan tool drags the stage natively ‚Äî the Select tool's drag is the marquee, so panning while it's active uses the same Space-hold / middle-drag path every other tool already had.
- Delete removes the whole selection (`handleDeleteLayers` filters out the background and locked layers rather than special-casing each call site).
- **Only text layers participate.** They're the only layers with Konva nodes and an `x`/`y` in the data model ‚Äî see "Move tool only repositions/relocates text layers" in Known gaps. (The Move tool's other job, dragging pixel *content* trapped inside a selection on a clean-patch layer, is a separate mechanism ‚Äî see Selection model above ‚Äî and isn't gated on this data-model gap, since it redraws the layer's own canvas rather than moving a Konva node's `x`/`y`.)

Two things here are non-obvious and were both found by testing rather than reasoning:

- **Konva's Transformer already moves every attached node** when a multi-selection is dragged, and it derives the combined bounding box from its node list. Propagating the drag delta to the other selected nodes by hand double-applies it and the followers overshoot. Each node just commits its own position in `onDragEnd`.
- A text layer's hit area is a `Rect`, and so is the page's drop-shadow backing ‚Äî the stage's "did I click empty canvas?" test goes by class name, so the hit rect carries `TEXT_HIT_NAME` and is excluded explicitly. Without that, clicking text counted as clicking background and cleared the selection the click had just made.

A finished marquee drag also fires a trailing stage click that would clear the selection it just made; `objectMarqueeConsumedRef` swallows it, mirroring `textDragConsumedRef`'s handling of the same problem for the Text tool.

A text layer's `draggable` prop excludes `spaceDown`, matching the Stage's own draggable calc ‚Äî without that, holding Space to pan while dragging a text layer fired **both** Konva's native node-drag and the manual space-drag pan loop from the same gesture, reading as the canvas jumping under the text mid-drag.

### Text layers (`TextLayerNode.tsx`, `textLayout.ts`, `textRuns.ts`, `TextPanel.tsx`, `textGradient.ts`)

A text layer is a layer-wide style **plus** `runs: TextRun[]` ‚Äî per-character overrides. Photoshop's split decides what lives where, and it's the same split ag-psd's `styleRuns` uses:

- **Per-character (`TextRun`)**: font family, weight, bold/italic, size, colour, tracking, kerning, baseline shift.
- **Layer-level**: stroke, shadow/glow, gradient (layer *effects*), and align/lineHeight/wrap (paragraph properties).

Two modules own the invariants, and nothing else may duplicate them:
- **`textRuns.ts`** owns the run invariant ‚Äî runs are contiguous and cover `content` exactly. Never hand-edit `runs`; go through `normalizeRuns`/`applyToRange`. Plain text is `[]`: a lone override-free run carries nothing the layer style doesn't, so it collapses (adjacent identical runs merge too, or the array fragments to one run per character after a few edits). `resolveRunStyle` is the **only** place default-vs-override precedence lives.
- **`textLayout.ts`**'s `layoutText()` owns wrapping, the box, alignment, baselines and run positions, and is shared by the canvas *and* the raster exporter. They each used to wrap independently, which is exactly how point text ended up exporting wrapped when it never wraps on screen.

**Point vs area (box) text**: `TextLayerData.autoWidth` ‚Äî point text grows to fit and never wraps; area/box text wraps at its authored `width`. Both share one Transformer with all 8 resize handles + a rotation handle (`StudioCanvas.tsx`); a corner drag scales `width` (reflow, via `layoutText`) and `fontSize` (glyph size) together rather than baking a Konva scale onto the node, so the layer's stored data stays the source of truth after every resize. Double-clicking a handle auto-fits a box layer's `width` tightly to its content (`layoutText({...text, autoWidth: true}).width`); triple-clicking inside editing text selects all (native textarea `select()`, gated on `e.detail === 3`). Area text can optionally set `fixedHeight`, in which case content that would overflow it shows a small ‚äû indicator (`layout.overflowing`) rather than clipping silently. Freshly-typed content that reads as majority-Arabic-script (`textLayout.ts`'s `isArabicMajority`) defaults a still-empty layer to right alignment the moment it's typed ‚Äî a one-time default on the empty‚Üínon-empty transition only, so it never fights a later manual change.

**Per-line overrides are a third styling tier**, above per-character runs: `TextLayerData.lineOverrides?: Record<number, LineStyleOverride>`, keyed by a line's index in the *current wrapped layout* (so a reflow that shifts line counts can shift what an override applies to ‚Äî an accepted trade-off since line overrides are an occasional balancing tool, not something edited every keystroke the way runs are). `textRuns.ts`'s `resolveLineStyle` applies on top of a run's already-resolved style ‚Äî line wins over run wins over layer default. Click a line on canvas (only while the layer is selected and not being edited ‚Äî independent of the editing textarea's own character-range selection) to target it; `TextLayerNode.tsx` renders one invisible hit `Rect` per `layout.lines` entry. `layoutText()` re-measures an overridden line's pieces (a line override can change `fontSize`/`letterSpacing`, which the pre-wrap widths used for every *other* line's alignment don't reflect) so its own wrap width/alignment stay self-consistent. Flows through PNG/JPG/WEBP export for free via the shared `layoutText` path; **PSD's `styleRuns` are per-character only**, so line overrides don't carry into PSD export yet.

**Konva has no rich text** ‚Äî `Konva.Text` renders one uniform style. So `TextLayerNode.tsx` renders a `Group` with **one `Konva.Text` per positioned run**, placed by `layoutText`; Konva still paints the glyphs, we only position them. A custom `sceneFunc` would have meant reimplementing stroke/shadow/gradient painting. The Group is the interactive/transform target and carries a `transparent`-filled `Rect` for hit area and bounds (a transparent fill still reaches the hit canvas, where Konva substitutes its own colour key), so clicking gaps between glyphs selects the layer and the Transformer frames the text box rather than glyph extents.

Load-bearing details:
- `measureRunText` deliberately uses **Konva's own formula** (`measureText(text).width + letterSpacing * length`, measuring *without* native `ctx.letterSpacing`) ‚Äî Konva draws spaced text letter-by-letter with that advance, so matching it is what keeps a run's laid-out width equal to the width Konva renders, which is what lets runs butt up against each other.
- **Kerning is modelled as extra advance *before* a run.** A manual kern is applied at a caret position between two characters, and a caret position is exactly a run boundary ‚Äî that's what makes it faithful rather than an approximation.
- Line height comes from the **layer's** `fontSize`, so a single big run doesn't reflow the whole block (matching what `lineHeight` has always meant here). Runs share a baseline, so a bigger run's em box starts higher.
- Konva's `fillPriority` defaults to `'color'`, so a node with both `fill` and gradient stops silently renders the **flat colour**; it must be set to `'linear-gradient'` when a gradient is on.

**Gradient text is real** (`TextLayerData.gradient`). `textGradient.ts`'s `gradientVector(w, h, angleDeg)` is shared by canvas and exporter; it *projects* the box onto the gradient direction rather than running corner-to-corner, so the ramp spans the full box at any angle. Angle is degrees clockwise from left-to-right in **screen coords (y down)**, so 90¬∞ = top-to-bottom. The ramp is computed once on the **layout box** and each run subtracts its own offset (Konva gradient points are node-local) ‚Äî without that subtraction every run would restart the ramp. PSD carries the gradient as a `gradientOverlay` **effect** (how it's done by hand in Photoshop, and still editable there), with `style.fillColor` left as the flat colour underneath. ag-psd normalizes gradient stop `location`/`midpoint`/`opacity` to **0..1** and scales them on write ‚Äî they are *not* raw 0..4096 PSD units. Photoshop's angle is counter-clockwise/y-up, so `exportPsd` negates ours.

**Character selection** is the editing `<textarea>`'s own `selectionStart`/`selectionEnd` ‚Äî there's no separate selection model. It's lifted to `Studio.tsx`'s `textSelection` so `TextPanel` (a dock sibling) can style it. With a range selected the character controls write run overrides; with no range they apply layer-wide (what Photoshop's Character panel does when the layer rather than a range is selected) ‚Äî and a layer-wide change also *clears* that key's run overrides, or the new default would be masked by runs and the control would look broken. Triple-click-to-select-all is handled on the textarea, not the Konva node: once editing starts the textarea sits above the stage, so the third click never reaches the shape.

`reflowRunsForContent` re-anchors runs when `content` changes, diffing by common prefix/suffix ‚Äî the textarea only hands us the new string, not an edit operation, so without it styled spans would slide off their characters as you type.

**Character and paragraph styles** (`src/lib/textStyleStore.ts`) apply at **layer level**, not to a selection ‚Äî `CHARACTER_STYLE_KEYS` includes layer-only fields (shadow/gradient/stroke), so routing them through runs would half-apply them.

**Saved character and paragraph styles are real** (`src/lib/textStyleStore.ts`, idb-keyval `text_styles`, schema-versioned). A character style captures appearance (font/size/colour/bold/italic/tracking/stroke/shadow/gradient); a paragraph style captures layout (align/lineHeight/autoWidth). `CHARACTER_STYLE_KEYS`/`PARAGRAPH_STYLE_KEYS` are the single source of truth for both capture and apply ‚Äî applying one touches **only** its own subset, which is the whole point of the split (restyle layout without overwriting the font). Capture and apply both `structuredClone`, so a saved style never shares its nested `shadow`/`gradient` object with a layer.

### TypeR (scripted lettering)

`studioTypes.ts`'s `parseTyperScript()` + `TyperPanel.tsx`. Paste a script, arm it, click bubbles to stamp lines in order with per-line styles matched by prefix (longest prefix wins; empty-prefix style is the catch-all).

Ported from the real TypeR 2.5 extension's documented behavior (see `src/assets/claude.md` for the original algorithm description, and `src/assets/mangastudio (1).html` for a working reference implementation):
- `##`-prefixed lines are ignored (notes).
- `//`-prefixed lines continue (append to) the previously placed line rather than starting a new one.
- `Page N` control lines (English or Arabic, incl. Arabic-Indic digits) tag the next real line with a page hint; `Studio.tsx` auto-switches pages as the armed script advances onto a hinted line (matches by number extracted from the target page's filename, falling back to 1-based position).
- A line fully wrapped in `**bold**`/`<b>...</b>` or `*italic*`/`<i>...</i>` overrides that style's weight for just that placement.
- Styles have an optional `folder` field for organizing the panel into collapsible groups ‚Äî **folder does not affect prefix-matching priority** (simplified from the original "focused folder" UI concept, which doesn't exist here); matching is purely longest-prefix-first across all styles.

**Multi-Bubble mode is real**: toggle it on in the TypeR panel (a "Layers"-icon button next to Arm), and arming switches the active tool to Rectangular Marquee instead of Text. Draw a rect per bubble and click "Add Bubble" to queue it (rendered as a distinct amber dashed overlay, `StudioCanvas`'s `queuedBubbleRects` prop, alongside the normal white live-selection outline) ‚Äî then "Place All" stamps each queued rect's line, in script order, centered in that rect, in one `updateLayers` call (`Studio.tsx`'s `handleAddBubbleRect`/`handlePlaceAllBubbles`). **Not implemented**: per-character rich text *from a script* ‚Äî bold/italic overrides still apply to the whole placed layer. `TextLayerData` now has a run model (see Text layers above), so this is no longer blocked on the data shape; it's that `parseTyperScript` only marks a whole line bold/italic, so nothing emits partial runs.

**Type Region is a separate, freehand extension of TypeR ‚Äî not a replacement for the panel above.** The panel's own arm state and its `placeInSel`-equivalent/`mqAuto`-equivalent/`mbFill`-equivalent flow (`handleAddTextLayer`'s TypeR branch, `handleAddBubbleRect`/`handlePlaceAllBubbles`, `parseTyperScript`) are untouched by it. Type Region has its own arm toggle (`Studio.tsx`'s `typeRegionArmed`, a `StudioToolbar` button, deliberately no keyboard shortcut so it can't collide with the panel's own) and a different job: turning a selection made with *any* selection tool into a live text container, immediately, for direct freehand typesetting rather than a scripted pass. While armed, `StudioCanvas.tsx`:
- Intercepts a pointerdown that lands inside an existing selection (`selectionContainsPoint`), ahead of whatever the currently active tool would otherwise do with that click, and hands the shape to `Studio.tsx`'s `handleCreateTypeRegion`.
- Also auto-converts the instant a **new** Rectangular/Elliptical Marquee or freehand Lasso selection completes (not row/col/Crop/Slice, and not yet wand/pen ‚Äî those still only work via the click-inside-an-existing-selection path above) ‚Äî one continuous gesture, no separate "Add Bubble" click.

`handleCreateTypeRegion` sizes the new text layer to `paint/selection.ts`'s `selectionBounds(shape)` (any selection kind's bounding box, computed directly for rect/ellipse/polygon rather than rasterizing, since `mask` already carries its own). A non-rectangular shape also gets a `TextLayerData.clipShape` (`{kind:'ellipse'|'polygon', ...}` in the same image-space coords as `x`/`y`) ‚Äî a rect needs no clip (the frame itself already is the rect), and a `mask`-kind selection has no clean vector form to persist, so it falls back to a plain rectangular container at its bounds instead of storing anything. `clipShape` drives a Konva `clipFunc` on the text layer's `Group` (`TextLayerNode.tsx`, translating the stored page-space shape into the Group's local space by subtracting `text.x`/`y`) and the identical shape clips `exportImage.ts`'s flattened PNG/JPG/WEBP export (there, `ctx` is still in page-space at that point, so no translation is needed). The shape's dashed outline is a **sibling** of the main text `Group`, not a child of it ‚Äî the main Group is `visible={false}` while editing, which is exactly when the outline needs to stay visible so the container's real shape reads behind the editing textarea. `pathForSelection` is exported from `paint/selection.ts` (previously module-private) so both the paint-clip path and this one build the exact same `Path2D`. **PSD export doesn't carry `clipShape`** ‚Äî `ag-psd`'s vector-mask mapping wasn't attempted for this; a Type Region text layer round-trips through PSD as an unclipped rectangular text layer, still editable.

### Dock/panel system (`dock/`)

There is no region/floating-panel system (no `dockLayout.ts`, no `FloatingPanel.tsx` ‚Äî an earlier draft of this doc described an aspirational one). What's real: `DockContext.tsx` tracks one `activeTab: string | null` for a single tab strip, persisted to `localStorage` under `studio_right_panel_<chapterId>`, debounced ‚Äî scoped per chapter so switching projects doesn't bleed one chapter's panel arrangement into another's. `RightDock.tsx` renders that strip (Text/Adjustment/TypeR/Translation/Brushes/Fonts/History/Pages ‚Äî only one visible at a time).

**Color and Layers are *not* part of that swappable strip** ‚Äî they're two pinned, always-visible sections that bookend it in the right column (`Studio.tsx`'s `toolsSidebar`: Color on top, the `RightDock` strip in the middle, Layers on the bottom), each wrapped in `dock/CollapsibleDockSection.tsx` with its own chevron-collapse state (`colorPanelCollapsed`/`layersPanelCollapsed`) that shrinks it to just its header ‚Äî never fully hides it, and collapsing one hands its space to whichever sections are still expanded. This exists because the old single-shared-strip design meant switching to *any* other tab (most commonly Text/Adjustment, auto-triggered by selecting a layer ‚Äî see the Layers panel paragraph below) hid the Layers list entirely, on the very click meant to just select a row in it. Same three-section structure on the phone bottom sheet. The Window menu's per-panel show/toggle entries route 'color'/'layers' to these two collapse states instead of `dock.selectTab`.

**Selecting a layer never switches `dock.activeTab`.** `selectLayer`/`selectLayers` (`Studio.tsx`) used to call `dock.selectTab('text'|'adjustment')` on every select, including a plain single click ‚Äî since Text/Adjustment lived in the one shared strip Layers also used to share, that hid the Layers list on the very click meant to select a row in it. Now selecting only selects; opening a row's settings (the existing opacity/blend/mask/clip/duplicate/move/delete strip, plus rename) is a separate, explicit gesture ‚Äî double-click the row or its hover-revealed ‚öô (`LayersPanel.tsx`, `onToggleExpanded`, unchanged from before ‚Äî only what triggers it changed) ‚Äî so the list itself never disappears. Each row also gets a hover-revealed (always-on for touch, via a `[@media(pointer:coarse)]` variant) trash icon for a one-click delete without expanding first, and a right-click context menu (Select/Rename/Duplicate/Hide-Show/Lock/Move Up/Move Down/Delete) built the same way the app's other ad hoc menus are ‚Äî a `swal` `input:'select'` dialog (`BrushesPanel.tsx`'s preset menu uses the identical pattern) ‚Äî rather than a new floating-menu component. Multi-select (Ctrl/Shift-click) and undoable multi-delete (`Delete` key ‚Üí `handleDeleteLayers(selectedLayerIds)`) were already correct and didn't need to change.

Responsive breakpoints (`Studio.tsx`'s `layoutMode`): `desktop` (‚â•1024px, side dock), `tablet` (768‚Äì1024px, dock collapses to a tap-to-open icon strip overlay), `phone` (<768px, dock opens as a near-full-height slide-in sheet ‚Äî `animate-slide-up-sheet` in `index.css` ‚Äî with a drag-handle/close affordance, not a fixed-height strip). Tool rail is already icon-only/horizontal on phone via existing `lg:hidden` breakpoint classes; StudioToolbar serves as the top mini-bar on every breakpoint.

Fullscreen: native Fullscreen API on the Studio root, synced to `fullscreenchange` (not just button state, so Esc/browser UI stays in sync). Not bound to literal F11 ‚Äî browsers intercept that key at the chrome level before JS reliably sees it ‚Äî bound to `Ctrl/Cmd+Shift+F` instead. `Tab` hides all panels (menu bar, tool options, tool rail, dock, floating panels), guarded against firing while any input/textarea/contenteditable has focus.

Window menu items render real checkmarks (`Menu.tsx`'s `checked` field) reflecting actual panel/fullscreen/hidden state ‚Äî not decorative.

## Text Editor (`src/components/textEditor/TextEditorPage.tsx`)

A standalone top-level page (`src/config/navTabs.ts`'s `'text-editor'` nav tab), **not** embedded in Studio ‚Äî reached via the sidebar/bottom nav, matching the "own page, own switcher" requirement. TypeR itself is Studio-only; the editor connects to it one-way via a "Send to TypeR" pipeline (see below).

- **Multi-document tabs**: `TextEditorDoc[]` (`src/lib/textEditorStore.ts`), each with its own pages, persisted to IndexedDB (`text_editor_docs` key, separate from both Studio stores).
- **Pagination**: each doc's pages are A4-sized (`794√ó1123px`) `contentEditable` divs. `reflow()` does direct DOM manipulation on refs ‚Äî pushes overflowing trailing block-children to the next page, pulls blocks back up from the next page to fill gaps ‚Äî and only touches React state to change *page count* (splice the array, keep every untouched page's string reference identical). This is deliberate: re-rendering a page's `dangerouslySetInnerHTML` on every keystroke would reset that page's DOM and kill the caret mid-typing, so normal content edits never flow back through state ‚Äî only autosave (on a debounce timer, captured fresh from refs) and explicit bulk actions (spell-check apply, find/replace, doc switch) do, and those intentionally force a full page re-render via `renderKey`.
- **Formatting**: `document.execCommand`-based toolbar (bold/italic/underline/headings/lists/align) ‚Äî the same pragmatic approach the reference prototype used; no custom rich-text model.
- **Spell-check**: `src/lib/spellCheck.ts` ‚Äî a small starter EN+AR misspelling dictionary (not a full language dictionary ‚Äî there's no server to back a real one), flags exact-word matches as clickable `.spell-miss` spans, click-to-fix. `stripSpellMarks()` always runs before export/send, matching the reference prototype's "export strips spell marks" convention.
- **Export**: TXT (real, `pageToPlainText`), DOCX (real, via the `docx` npm package ‚Äî dynamically imported so it doesn't bloat the main bundle, block-level HTML‚Üíparagraph/run mapping with bold/italic/underline/heading levels preserved), PDF (`printDocAsPdf` opens a print-formatted window with `@media print` A4 page breaks exactly matching the on-screen pages, then calls `window.print()` ‚Äî deliberately avoids a heavy client-side PDF-rendering library since the browser's native print-to-PDF already produces print-quality, pixel-accurate output).
- **Send to TypeR**: strips spell marks, joins all pages' plain text, and calls `onSendToTyper` ‚Üí `App.tsx`'s `pendingTyperScript` state ‚Üí `Studio.tsx` consumes it into `typerScript` on mount and clears it. This means the text is "waiting" in whichever chapter's Studio the user opens next, without either page needing to be mounted simultaneously.
- **Not implemented this pass**: tables, images, comments, track changes, and the *optional* in-editor floating panels (Translation Preview and a live read-only page thumbnail) that SPEC describes as opt-in helpers while writing ‚Äî real, separate scope beyond a first working editor. No dedicated "switch Studio ‚Üî Editor" keyboard shortcut either; switching is one click via the existing nav rail/tab bar.

## Color system (`color/ColorPanel.tsx`)

RGB/Hex/HSV wheel + recent colors were already real; HSL and CMYK rows now use `colord` (+ its `cmyk` plugin, `extend([cmykPlugin])`) for the conversions rather than hand-rolled math ‚Äî small, well-tested library, not bundled-heavy (a few KB, stays in the main chunk). **Saved palettes are real** ‚Äî `src/lib/paletteStore.ts` (idb-keyval, `color_palettes` key, mirrors `fontsStore.ts`'s pattern) backs named `ColorPalette` lists in `ColorContext`; the panel's Palettes section lets you create/delete a palette and add/remove swatches (right-click a swatch to remove it). **Not implemented**: a dedicated multi-stop gradient editor UI (the Gradient *tool* itself is real and goes foreground‚Üíbackground, see Canvas engine above ‚Äî there's just no UI for arbitrary custom gradient stops beyond that one pair).

## Fonts (`FontsPanel.tsx`, `src/lib/fontLoader.ts`, `src/lib/fontsStore.ts`)

Real font installation: upload TTF/OTF/WOFF/WOFF2 ‚Üí `opentype.js` (dynamically imported) parses the embedded family name ‚Üí registered via the `FontFace` API (`document.fonts.add`) ‚Üí persisted to IndexedDB (`custom_fonts` key) and re-registered on every load. Installed families flow into `Studio.tsx`'s `allFontFamilies` (built-ins + custom) and are passed to both `TextPanel` and `TyperPanel` as a `fontFamilies` prop (defaults to the built-in list if omitted) ‚Äî TypeR styles have always had a `fontFamily` field, and the style editor's Font select now actually drives it. **Google Fonts integration is intentionally not implemented** ‚Äî SPEC lists it as optional, and pulling from Google's CDN would require network access, conflicting with this app's offline-capable design intent.

## Known gaps (honest, not silently dropped)

- **No Home/landing page.** Top nav is 5 flat tabs (Library/Text Editor/Settings/Teams/Cloud); Recent/Templates/Tutorials/Plugins/Account don't exist. Building nav destinations with no real content behind them would violate the no-placeholder rule ‚Äî needs real features first.
- **Interactive control sizing** on mobile/tablet breakpoints hasn't had a systematic ‚â•44px touch-target pass ‚Äî many buttons/sliders are still 24‚Äì32px (a few new additions, like the tablet dock icon strip, were sized to 44px; the sweep across every existing control wasn't done).
- **Smart objects** (layer *groups*, stack-aware adjustment layers, and layer *masks* all *are* implemented ‚Äî see Per-page layer stack above) ‚Äî declared as an intended feature (in tool lists, type unions, or SPEC) but not implemented. (Magnetic lasso, patch tool, slice tool, healing brush, and the full vector Pen Tool ‚Äî anchors/handles, Path Selection, Direct Selection, convert-anchor-point, Stroke/Fill Path ‚Äî *are* all implemented, see Canvas engine / Selection model / Vector paths above.) `toolGroups.ts`'s `curvature-pen` still shares an icon with plain `pen` ‚Äî matches Photoshop's own flyout, which has no separate glyph for it either, so this is intentional, not a gap.
- (Selection feather/expand/contract/add-subtract/intersect, Transform Selection, Quick Mask, liquify's `reconstruct` mode, Crop, TypeR's Multi-Bubble mode, and brush symmetry *are* implemented ‚Äî see Canvas engine / Selection model / TypeR above. The Layers panel navigating away on a plain click, and Color/Layers sharing a slot with other panels, *are* both fixed too ‚Äî see Dock/panel system above.)
- **Move tool only repositions (moves the Konva node of) text layers**, and for the same reason **canvas multi-select only covers text layers** (see Canvas selection above). Clean-patch (raster) and adjustment layers have no `x`/`y` offset in the data model and their Konva nodes aren't draggable ‚Äî real Move-tool *layer* support for them would need per-layer offset fields plus a Group wrapper (and matching changes to export/flatten), a separate piece of work like layer groups/masks above. (The Move tool *does* already drag pixel **content** within an active selection on a clean-patch layer, by redrawing the layer's own canvas rather than moving a node ‚Äî see Selection model above; that's a different mechanism and isn't blocked on this gap.)
- **Text warp** ‚Äî needs path-based glyph rendering, which Konva's `Text` has no support for; it would mean generating glyph outlines (via `opentype.js`, already a dependency) and deforming them, rather than positioning text nodes. Not blocked on the run model ‚Äî that now exists (see Text layers above), so kerning, baseline shift and per-character styling are implemented.
- **Vertical text mode (Japanese)** ‚Äî not implemented; `layoutText()` only lays out horizontal lines. Per-line overrides' PSD export is a separate, smaller gap: `exportPsd.ts`'s `styleRuns` are per-character only, so `TextLayerData.lineOverrides` doesn't carry into PSD (PNG/JPG/WEBP does, via the shared `layoutText` path).
- **Type Region's auto-bubble-on-new-selection only covers Rectangular/Elliptical Marquee and freehand Lasso.** Magic Wand and pen-path (Make Selection from Path) selections still become text containers via the click-inside-an-existing-selection gesture, just not the instant-on-completion one. A `mask`-kind selection (magic wand, Quick Mask output) has no clean vector form to persist as `TextLayerData.clipShape`, so Type Region falls back to a plain rectangular container at its bounds rather than a shape-clipped one; PSD export doesn't carry `clipShape` at all yet (see Text layers above).
- **TypeR's bold/italic overrides are still whole-layer.** The run model now makes per-character TypeR styling possible, but `parseTyperScript` only marks a *whole line* bold/italic, so nothing currently produces partial runs from a script ‚Äî wiring that up is a separate change to the parser and its placement path.
- **Color**: no multi-stop gradient editor UI (saved palettes *are* implemented, see Color system above; gradient *text* is real but is a two-stop from‚Üíto ramp, same as the Gradient tool). **Fonts**: no Google Fonts (by design, offline-first; TypeR style font-family control *is* implemented, see Fonts above). **Music player, navigator/minimap, safe-area overlay** ‚Äî not implemented (Grid and Rulers *are* implemented, see Canvas engine above).
- **Text editor**: no tables/images/comments/track-changes, no in-editor floating Translation-Preview/page-thumbnail panels, no dedicated Studio‚ÜîEditor shortcut.

## Conventions

- Keep the two IndexedDB stores (`workspaces_library` vs `studio_<chapterId>`) separate ‚Äî don't fold raster/layer data into the main library object; that was a deliberate autosave-cost decision.
- New persisted shapes should include a `schemaVersion` field and a real migration path (see `migrate.ts` for the pattern), not just hope old data happens to still parse.
- Large/optional dependencies (`ag-psd`, `docx`, `opentype.js`) are dynamically imported (`await import(...)`), not top-level, to keep the main bundle lean ‚Äî check `npm run build`'s chunk output after adding a new one; it should show up as its own chunk, not inflate the main one. Small libraries (`colord`, a few KB) are fine as regular top-level imports.
- `npm run lint` is actually `tsc --noEmit` (no separate linter configured) ‚Äî run it after any change; `npm run build` is the stronger check since Vite/Rollup catch a few things `tsc --noEmit` alone doesn't.
- **Tests**: `npm test` is vitest over `src/**/*.test.ts` ‚Äî pure logic only, no DOM. `npm run test:e2e` is Playwright (`e2e/`), driving the real app in Chromium; `vitest.config.ts` excludes `e2e/` or vitest would collect those specs and choke on Playwright's API. **e2e runs against a `vite build` + `vite preview`, never the dev server** ‚Äî HMR would reload the page mid-test whenever a source file changed, producing failures that aren't real and blocking any work during a ~15-minute run. The build cost is worth it, and it means the suite tests what ships. Reach for e2e whenever the thing you changed only exists on the Konva canvas ‚Äî **glyphs, strokes and layer compositing are not in the DOM**, so no DOM assertion can see them. `e2e/studioFixture.ts` has the helpers: `openStudio` (seeds a chapter into IndexedDB and clicks in) and `countDarkPixels` (a stage pixel probe).
- **Two traps the e2e pixel probe already fell into**, both of which made a test pass while proving nothing: (1) scope canvas queries to `.konvajs-content` ‚Äî the Brushes panel draws its presets in black on their own canvases and swamps any global scan; (2) assert **deltas**, not absolutes ‚Äî the stage carries a standing ~4.4k dark pixels from the page drop-shadow, so `> 0` is always true. If you write a regression test, verify it actually fails with the bug reintroduced before trusting it.
- The **whole app is gated behind `AuthGate`** (Supabase session) and `supabaseClient.ts` calls `createClient` at module scope ‚Äî so a missing `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` throws on boot and renders a blank page, not just a broken Teams tab. `playwright.config.ts` injects dummy values and the fixture seeds a fake session into localStorage; for local dev you need a real `.env.local`.
- This session ported algorithms/behavior from two references: `src/assets/mangastudio (1).html` (a disconnected single-file prototype ‚Äî TypeR parsing model, Tab-hide-panels, teReflow pagination) and the documentation in `src/assets/claude.md` (Arabic, describes that prototype's contracts). Both are read-only reference material, not part of the shipping app ‚Äî don't wire them in or assume their code paths exist here.
